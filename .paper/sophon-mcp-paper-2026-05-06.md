## 概要 (Abstract)

LLMを用いたAIコーディングにおいて、コンテキストウィンドウへのトークン消費はコスト・精度・速度に直結する課題である。本稿では、この問題を解決するために設計されたMCPサーバー「Sophon MCP」の全機能を、実際のNext.jsプロジェクトに対して検証し、標準ツール（ファイル全体読み取り）とのトークン使用量を定量的に比較した。

**主要な発見:**

- 大規模JSONファイル（package-lock.json）では **92.6%** のトークン削減を達成
- キーワード検索では最大 **96.4%** 削減
- 複合ワークフロー（プロジェクト初期調査）では **49.4%** 削減
- 小規模ファイルではJSON構造オーバーヘッドにより逆効果となるケースあり
- `export default function` 構文がパーサーに未対応であったバグは v1.0.1 にて修正済み（Next.js完全対応）

## 1. はじめに

### 1.1 背景

Claude Code等のAIコーディングツールは、ファイルを読む際に標準の `Read File` ツールを使用する。このアプローチは実装が単純である反面、数万トークンのファイル（例: `package-lock.json`）をそのままコンテキストに流し込むため、無駄なトークン消費を引き起こす。

コンテキストウィンドウの制限（Claude Sonnet 4.6: 200,000トークン）において、不要なトークン消費は以下の問題を引き起こす。

1. **コスト増加** — 入力トークンが多いほど課金額が増加する
2. **精度低下** — 関係のない情報がコンテキストを圧迫し、注意機構の焦点が拡散する
3. **速度低下** — 処理するトークン量に比例してレスポンスが遅くなる

### 1.2 Sophon MCPの設計思想

Sophon（ギリシャ語「賢明なもの」）は「必要な情報だけを必要なタイミングで渡す」という設計思想のもと構築された。具体的には：

- ファイル全体ではなく構造（目次・スケルトン・キー一覧）を先に返す
- AIが必要な部分を特定してから、その部分のみを取得する
- 全レスポンスに `token_count` を付与し、効果を可視化する

### 1.3 検証の目的

本稿の目的は以下の3点である。

1. Sophon MCPの全ツールを実環境で動作確認する
2. 各ツールのトークン削減効果を定量的に測定する
3. 効果的なユースケースと限界を明らかにする

## 2. 検証環境・方法

### 2.1 対象プロジェクト

```
D:\Dev\Sophon-TEST\nextjs-target\
├── app/
│   ├── layout.tsx    (183 tokens)
│   ├── page.tsx      (674 tokens)
│   └── utils.ts      (836 tokens) ← 検証用カスタムモジュール

├── package-lock.json (88,759 tokens)
├── tsconfig.json     (205 tokens)
└── README.md         (360 tokens)
```

`utils.ts` はbenchmark検証用として作成したTypeScriptユーティリティモジュールで、以下を含む：
- 単独エクスポート関数 × 5 (`formatDate`, `truncateText`, `fetchWithRetry`, `parseQueryString`, `deepMerge`)
- クラス定義 × 1 (`UserService` — 6メソッド含む)
- インターフェース定義 × 2

### 2.2 測定方法

```
標準アプローチ: fs.readFileSync(filepath) → 全内容をカウント
Sophonアプローチ: 各ツールの実際の返り値をカウント
トークン計算: gpt-tokenizer (cl100k_base) — sophon内部と同一
```

各テストで以下を記録する：

- `standardTokens`: 標準ツール（ファイル全体読み取り）のトークン数
- `sophonTokens`: Sophonツールの返り値のトークン数
- `reduction (%)`: `(standard - sophon) / standard × 100`

## 3. 実験結果

### 3.1 ディレクトリ探索 (`read_directory_tree`)

| テスト | 標準 | Sophon | 増減 |
|--------|------|--------|------|
| ディレクトリツリー (depth=3) | 95t | 156t | **−61t (−64.2%)** |

**結果:** create-next-appの小規模プロジェクト（13ファイル）では、フォーマット用の罫線文字（`├──`、`└──`、`│`）と構造のオーバーヘッドにより、シンプルなファイルパス一覧より多いトークンを消費した。

**考察:** `read_directory_tree` は大規模プロジェクト（100ファイル以上）で効果を発揮する。`node_modules`（数万ファイル）を自動除外することで、標準の再帰リストと比べて大幅に削減できる。

---

### 3.2 Markdownファイル読取

| テスト | 標準 | Sophon | 削減率 |
|--------|------|--------|--------|
| README.md — TOCのみ | 360t | 91t | **+74.7%** |
| README.md — TOC + 上位2セクション | 360t | 115t | **+68.1%** |
| README.md — TOC + 全セクション | 360t | 593t | −64.7% |

**結果:** TOC取得のみ・部分セクション取得では74.7〜68.1%の削減を達成。全セクションを取得すると、JSONラッパーのオーバーヘッドで逆効果となる。

**最適ユースケース:** 大きなMarkdownドキュメントから特定の情報（インストール手順、API仕様等）を探す際に有効。目次でセクションを特定してから必要なセクションのみ取得する二段階アプローチが最も効率的。

### 3.3 TypeScriptコード読取

対象: `app/utils.ts`（836トークン、5関数 + 1クラス(6メソッド) + 2インターフェース）

| テスト | 標準 | Sophon | 削減率 | 備考 |
|--------|------|--------|--------|------|
| スケルトンのみ | 836t | 487t | **+41.7%** | 12シンボル検出 |
| スケルトン + 2関数本体 | 836t | 598t | **+28.5%** | 487t + 111t |
| スケルトン + Classのみ | 836t | 757t | +9.4% | 最大コンポーネント取得 |

**スケルトン出力例（一部）:**

```json
[
  {"id":"func:21-25","type":"function","name":"formatDate","signature":"export function formatDate(date: Date, locale: string = \"ja-JP\"): string"},
  {"id":"class:70-116","type":"class","name":"UserService","signature":"export class UserService"},
  {"id":"method:78-80","type":"method","name":"findById","signature":"findById(id: string): User | undefined"},
  ...
]
```

**修正済みの制限事項（バグ）:** v1.0.0時点では `export default function Home()` などの `default` キーワードを含む関数がパーサーに未対応であることが判明していた。

```typescript
// v1.0.0では検出されなかったが、v1.0.1で対応済み
export default function Home() { ... }
export default function RootLayout({ children }: ...) { ... }
```

これはNext.jsのApp Routerの主要エクスポートパターンであり重大な問題であったが、**v1.0.1のパッチ配信により修正された** ことが確認されている。

### 3.4 JSON/YAMLファイル読取

| テスト | 標準 | Sophon | 削減率 | 備考 |
|--------|------|--------|--------|------|
| package.json — キー一覧 | 187t | 320t | −71.1% | 小ファイルで逆効果 |
| package.json — scriptsのみ | 187t | 29t | **+84.5%** | 値指定取得 |
| package-lock.json — キー一覧 (depth=2) | **88,759t** | **6,573t** | **+92.6%** | 大規模JSON |

**結果:** この実験で最も顕著な結果が得られた。`package-lock.json`（88,759トークン）に対してキー構造一覧を取得することで、92.6%削減（82,186トークン節約）を達成した。

**量的インパクト:** Claude Sonnet 4.6のAPIでは入力100万トークンあたり約$3.00の課金が発生する。`package-lock.json` を1回読む場合：
- 標準: 88,759t × $3.00/1M = **$0.266**
- Sophon: 6,573t × $3.00/1M = **$0.020**
- 1回の読み取りあたり **$0.246の節約**

### 3.5 ファイル検索 (`search_file`)

| テスト | 標準 | Sophon | 削減率 | マッチ数 |
|--------|------|--------|--------|----------|
| 'async'検索 (context=2) | 836t | 30t | **+96.4%** | 1件 |
| 'User'検索 (context=1) | 836t | 598t | +28.5% | 21件 |

**結果:** マッチ件数が少ないほど削減率が高い。'async' のように特定機能に紐づくキーワードでは96.4%削減を達成。'User' のように頻出するキーワードでは28.5%にとどまった。

**削減率の推定式:** マッチ数をM、コンテキスト行数をCとすると、返り値のサイズはおよそ `M × (1 + 2C)` 行に比例する。ファイル全体のトークンに比べてこれが小さいほど効果的である。

### 3.6 テキスト圧縮 (`compress_text`)

| テスト | 標準 | Sophon | 削減率 | 内訳 |
|--------|------|--------|--------|------|
| utils.ts (コメント+空白除去) | 836t | 709t | **+15.2%** | comments: 8t, whitespace: 88t |
| 大規模APIレスポンス (null除去) | 1,549t | 377t | **+75.7%** | null値・空配列除去 |

`compress_text` のデフォルトオプション：

| オプション | デフォルト | 効果 |
|------------|----------|------|
| `remove_markdown_noise` | true | `---`区切り・HTMLコメント・バッジ除去 |
| `remove_duplicate_lines` | true | 連続重複行の圧縮 |
| `normalize_whitespace` | true | 余分な空行・スペース除去 |
| `remove_code_comments` | false | コードコメント除去（要明示指定） |
| `trim_tool_response` | false | null値・空配列・長URL除去（要明示指定） |

**null除去の効果が顕著な理由:** ツールレスポンスのJSONには `null` や `[]` が多く含まれることが多く、`trim_tool_response: true` でこれらを除去することで大幅な圧縮が可能。

### 3.7 Git差分 (`read_git_diff`)

テスト環境: `app/layout.tsx` に1行コメント追加

| テスト | 標準 (raw) | Sophon | 増減 |
|--------|------------|--------|------|
| git diff (1行変更) | 90t | 201t | −111t (−123%) |
| git diff (compress=true) | 90t | 195t | −105t (−117%) |

**Sophonの出力フォーマット:**

```
## Summary
app/layout.tsx | 1 +
 1 file changed, 1 insertion(+)

diff --git a/app/layout.tsx ...
```

**結果:** 小さな差分ではstatサマリのオーバーヘッドにより逆効果となる。多ファイルにわたる大規模なPRでは、stat行が変更の全体像を把握するのに役立ち、コンテキスト行数削減（`-U3`）と合わせて削減効果が期待できる。

## 5. 発見された制限・バグ

### 5.1 `export default function` 未対応（v1.0.1で修正済み）

**現象:** v1.0.0において、Next.js App Routerの標準パターン（`export default function Page()`）がコードスケルトンパーサーに検出されないバグが存在した。

**原因と修正:** `read_code_skeleton.js` の正規表現が `default` キーワードを考慮していなかったため。v1.0.1のパッチにて以下の修正案が取り込まれ、解決された。

```javascript
// v1.0.0 のパターン
{ regex: /^(?:export\s+)?(?:async\s+)?function\s+(\w+)/, type: "function" }

// v1.0.1 で適用されたパターン
{ regex: /^(?:export\s+)?(?:default\s+)?(?:async\s+)?function\s+(\w+)/, type: "function" }
```

**結果:** `page.tsx` や `layout.tsx` などNext.jsの中核コンポーネントのスケルトンが正常に取得できるようになった。

### 5.2 ~~小規模ファイルでの逆効果~~（v1.1.1で自動フォールバック対応済み）

**v1.0.1時点の現象:** 200トークン以下の小ファイルでは、構造化レスポンスのJSON/テキストオーバーヘッドが本来の削減分を上回り、トークン数が増加していた。

**v1.1.1での修正内容:**

- `read_json_yaml_keys`: 300トークン未満のファイルは構造化せずそのまま `raw_content` として返すよう自動フォールバックを実装
- `read_directory_tree`: ファイル数が30未満の小規模ディレクトリでは `token_count: 0` を返し、利用者にコスト効果がないことを明示するよう変更

**残存する閾値の目安（git diff は未対応）:**
- git diff: 変更行数 ≥ 20行程度で有効（未修正）

### 5.3 Markdown全セクション取得の逆効果（重大度: 低）

**現象:** `read_markdown_section` で全アンカーを指定すると、JSONラッパーオーバーヘッドにより全体読み取りより多くなる。

**対策:** 必要なセクションのみ指定する（Sophon設計の本来の使い方）。

### 5.4 Git差分の統計オーバーヘッド（重大度: 低）

**現象:** 変更が少ない場合、statサマリ（`## Summary\n...`）の付加により総トークンが増加する。

**効果的なケース:** 10ファイル以上にわたる大規模PRで、全体変更規模をAIが素早く把握する用途。

## 6. メモリ・タスク機能の評価

### 6.1 memory機能（`memory_save/get/list/delete`）

**機能:** SQLiteを使ったプロジェクト単位のキーバリュー永続保存。

**トークン効率への貢献（間接効果）:**
同じ会話を再開した際やコンテキストリセット後に、前回の設計決定・注意事項を再読み込みするコストを削減できる。例えば、「このプロジェクトはTypeScript strict mode」「DBはPostgreSQL 15」などのメタ情報をmemoryに保存することで、README全文を再読するコストを回避できる。

### 6.2 task機能（`task_create/update/get/list/delete`）

**機能:** 親子階層対応のタスク管理をSQLiteで永続化。

**コンテキスト節約効果:** 長期・複数セッションにわたる作業で、タスク状態（TODO/in_progress/done）を`task_list`で確認することで、前回の作業内容を会話履歴に保持する必要がなくなる。

## 7. 考察

### 7.1 どのような場面でSophonが最も有効か

| シナリオ | 推定削減率 | 理由 |
|----------|-----------|------|
| 大規模依存管理JSONの確認 | 90%+ | 数万トークンのファイルを100以下に圧縮 |
| 特定機能の検索・特定 | 70〜96% | 必要な行だけ取得 |
| ドキュメントの特定セクション参照 | 50〜75% | TOC→セクションの二段階取得 |
| 新規プロジェクトの初期把握 | 40〜50% | 複数ファイルをまたぐスケルトン探索 |
| 大規模APIレスポンスの処理 | 50〜75% | null/空値除去 |

Sophon MCPはAIコーディングにおけるトークン消費削減ツールとして、特定のユースケースで顕著な効果を発揮することが実測により確認された。
